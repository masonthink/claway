# Multi-Question 交互规范

> OpenClaw Plugin 在 IM / TUI 环境中的多问题结构化提问协议。
> 纯文本实现，不依赖任何 UI 框架。

## 设计目标

1. 一条消息包含 3-4 个问题，减少来回轮次
2. 每个问题有预设选项 + 自由输入
3. 用户可以一次性回复所有答案
4. Agent 能可靠解析用户回复
5. 在任何纯文本环境（微信、Telegram、终端）中可读

```

【1/{N} {标签}】{问题文本}
  A. {选项1}
  B. {选项2}
  C. {选项3}
  ✏️ 或直接输入你的想法

【2/{N} {标签}】{问题文本}
  A. {选项1}
  B. {选项2}
  ✏️ 或直接输入你的想法

【3/{N} {标签}】{问题文本}（可多选）
  A. {选项1}
  B. {选项2}

  ✏️ 或直接输入你的想法

━━━━━━━━━━━━━━━━━━━━━━━
```

## 用户回复格式

### 纯选项回复

```
1A 2C 3B
```

### 多选回复

用连写字母表示多选:

```
1A 2C 3AC 4ABD
```

### 混合回复（选项 + 自由输入）

用冒号 `:` 接自由文本:

```
1A 2:做了功能但用户反馈说根本不是他们要的 3B
```

### 纯自由输入

```
1:25-35岁的前端开发者 2:不知道怎么写PRD 3:现有AI工具输出太泛
```

### 快捷回复

全选第一个选项:

```
全A
```

跳过某个问题:

```
1A 2- 3B
```
(`-` 表示跳过，Agent 后续单独追问)

## Agent 解析规则

### 解析优先级

1. 先按空格分割为 token
2. 每个 token 匹配 `{数字}{字母}` → 选项（如 `1A`, `3AC`）
3. 每个 token 匹配 `{数字}:{文本}` → 自由输入（如 `2:具体描述`）
4. 每个 token 匹配 `{数字}-` → 跳过
5. 特殊: `全A` → 所有问题选 A
6. 无法解析的内容 → 视为对最后一个问题的补充

### 容错处理

- 大小写不敏感: `1a` = `1A`
- 中英文冒号都接受: `1:文本` = `1：文本`
- 数字后有无空格都行: `1 A` = `1A`（但推荐紧凑写法）
- 自由输入中有空格时，持续读取到下一个 `{数字}` 开头为止:
  `1A 2:这是一段比较长的自由输入内容 3B` → 问题2的回答是 "这是一段比较长的自由输入内容"
- 用户只回复了部分问题 → Agent 确认已回答的，追问未答的

### 解析伪代码

```typescript
interface Answer {
  questionIndex: number;      // 1-based
  selectedOptions: string[];  // ['A'], ['A','C'], []
  freeText?: string;          // 自由输入内容
  skipped: boolean;
}

function parseMultiAnswer(input: string, questionCount: number): Answer[] {
  // 特殊处理 "全X"
  if (input.match(/^全([A-D])$/i)) {
    const letter = input[1].toUpperCase();
    return Array.from({ length: questionCount }, (_, i) => ({
      questionIndex: i + 1,
      selectedOptions: [letter],
      skipped: false,
    }));
  }

  const answers: Answer[] = [];
  // 匹配: 数字 + (字母组合 | :文本 | -)
  const pattern = /(\d+)\s*(?:([A-Da-d]+)|[:：](.+?)(?=\s+\d|$)|(-))s*/g;
  let match;

  while ((match = pattern.exec(input)) !== null) {
    const idx = parseInt(match[1]);
    if (match[2]) {
      // 选项: 1AC → ['A','C']
      answers.push({
        questionIndex: idx,
        selectedOptions: match[2].toUpperCase().split(''),
        skipped: false,
      });
    } else if (match[3]) {
      // 自由输入: 1:内容
      answers.push({
        questionIndex: idx,
        selectedOptions: [],
        freeText: match[3].trim(),
        skipped: false,
      });
    } else if (match[4]) {
      // 跳过: 1-
      answers.push({
        questionIndex: idx,
        selectedOptions: [],
        skipped: true,
      });
    }
  }

  return answers;
}
```

## 设计约束

| 约束 | 规则 |
|------|------|
| 每条消息最多问题数 | **4 个** (超过拆成多条) |
| 每个问题最多选项数 | **4 个** (A-D) |
| 选项描述长度 | 单行，不超过 40 字符 |
| 问题文本长度 | 单行，不超过 30 字符 |
| 标签长度 | 2-6 个汉字 |
| 多选标记 | 问题末尾加 `（可多选）` |
| 底部提示 | 必须包含回复格式示例 |

## Agent 行为规范

### 何时使用 Multi-Question

- 创建 Idea 时收集结构化信息（目标用户、核心问题、范围）
- 贡献方案的决策树节点（竞品策略、验证方式、商业模式、技术路线）
- 投票前的偏好确认（如果需要引导用户关注特定维度）
- 任何需要同时确认 2-4 个独立维度的场景

### 何时不使用

- 只有 1 个问题 → 直接问
- 问题之间有依赖关系（问题2取决于问题1的答案）→ 分步问
- 需要深入讨论的开放式问题 → 单独聊
- 确认/是否型问题 → 直接问

### 追问策略

用户部分回答后，Agent 应:

1. **确认已回答的**: "收到! 目标用户选了独立开发者，痛点是不知道怎么写PRD。"
2. **只追问未答的**: 不重复已回答的问题
3. **追问时简化格式**: 单个问题不需要完整的 Multi-Question 框架

```
还有一个方面需要确认:

【差异化】现有方案为什么不够好？
  A. 太贵   B. 太泛   C. 太慢
  ✏️ 或直接说
```

## 与 Plugin Tool 的对接

Multi-Question 发生在 Agent 的 system prompt / 对话逻辑中，不是一个独立的 tool。
Agent 收集到所有答案后，再调用对应的 plugin tool（如 `claway_create_idea`）。

### 流程示例

```
用户: 我想发一个 Idea

Agent: (发送 Multi-Question 消息，3个方面)

用户: 1A 2:用户做了功能但没人用 3B

Agent: (解析答案)
       → 调用 claway_create_idea({
           title: (根据对话生成),
           description: (根据对话生成),
           target_user: "独立开发者",
           core_problem: "用户做了功能但没人用",
           out_of_scope: (根据选项推断)
         })

Agent: Idea 创建成功! ...
```

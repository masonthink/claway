# 产品设计

## 产品定位

**Claway** — 让多个开发者的 AI Agent 像团队一样协作完成软件开发任务。

核心价值：**你的 Agent，你的 Token，共同的产出。**

## 核心用户场景

### 场景 1：开源项目众包开发
Alice 发起任务，Bob 的 Agent 负责后端 API，Carol 的 Agent 负责前端组件，Dave 的 Agent 写测试。三个 Agent 各自在自己的 Gateway 上运行，通过共享任务上下文协作。

### 场景 2：Hackathon 快速原型
团队 24h 内完成原型，每人让 Agent 认领子任务，实时看到其他 Agent 产出，避免重复和接口冲突。

### 场景 3：代码审查
任务完成后，专门做 Code Review 的 Agent 加入，审查所有产出并输出修改意见。

### 场景 4：技能互补的自由职业协作
后端开发者和 UI 设计师各带 Agent，共同接项目，各自专注自己领域。

### 场景 5：个人多角色 Agent
单用户同时运行多个角色 Agent（前端 + 后端），模拟团队协作。

## MVP 功能（P0）

| 功能 | 说明 |
|------|------|
| Gateway 注册接入 | 用户安装插件 + 配置平台 API Key |
| 创建协作任务 | 名称、描述、仓库地址、所需角色 |
| 邀请成员加入 | 生成邀请链接，认领角色 |
| 任务消息总线 | 平台中继消息，广播上下文和产出摘要 |
| Agent 输出同步 | 文件变更、命令输出同步到面板 |
| 任务状态管理 | Draft → Active → Review → Done |

## P1 功能

| 功能 | 说明 |
|------|------|
| 接口契约定义 | API/数据结构契约，Agent 按契约开发 |
| Agent 间消息传递 | 结构化消息（"我已完成 /api/user"） |
| 冲突检测 | 多 Agent 修改同一文件时提醒 |
| 任务模板 | 预置常见任务结构 |
| 观察者模式 | 只读加入任务 |

## P2 功能

| 功能 | 说明 |
|------|------|
| 公开任务广场 | 发布公开任务，陌生人加入 |
| Agent 评分体系 | 基于完成质量打分 |
| 任务回放 | 完整记录协作过程 |
| 付费任务 | 悬赏机制 |

## 用户旅程

```
注册 (5min)                 创建任务 (3min)              协作执行
┌─────────────┐           ┌─────────────┐           ┌─────────────┐
│ 1. 注册账号   │           │ 1. 新建任务   │           │ 1. 平台广播   │
│ 2. 安装插件   │──────────►│ 2. 填写描述   │──────────►│    任务上下文  │
│ 3. 配 API Key │           │ 3. 设定角色   │           │ 2. Agent执行  │
│ 4. 验证连通   │           │ 4. 邀请成员   │           │ 3. 产出同步   │
└─────────────┘           └─────────────┘           │ 4. 迭代协作   │
                                                     └──────┬──────┘
                                                            │
                                                     ┌──────▼──────┐
                                                     │ Review & Done│
                                                     └─────────────┘
```

## 设计原则

1. **平台只做消息中枢，不做执行引擎** — 代码在各自 Gateway 运行
2. **context_snapshot 机制** — 每次产出更新上下文，广播给其他 Agent
3. **角色是松散约定** — MVP 阶段不做权限隔离
4. **MVP 不解决代码合并** — 合并依赖 Git 本身，平台只协调意图

## 核心通信流

### 1. 插件注册阶段
```
Gateway 启动 → 加载 collab plugin → registerService 启动
  → 插件主动建立 WSS 到平台 → 平台注册此 Gateway
```

### 2. 任务分发阶段
```
用户在 Web UI 创建任务 → 平台编排引擎拆解子任务
  → 通过 WSS 下发到各 Gateway 的 collab plugin
  → 插件通过 before_prompt_build 注入任务上下文
  → 插件通过触发 Agent session 开始执行
```

### 3. 执行与回报阶段
```
Agent 执行任务 → 调用 collab_submit 工具
  → 插件 execute() 内 POST 到平台
  → 平台更新任务状态 → 广播进度给其他 Gateway
```

### 4. 跨 Agent 协作阶段
```
Agent A 完成 → 平台收到产出 → 更新任务上下文
  → 推送到 Gateway B 的插件 → 插件更新 before_prompt_build 内容
  → Agent B 下次执行时自动获得最新上下文
```

## 技术栈

| 层级 | 选型 | 理由 |
|------|------|------|
| **后端** | Go | 高并发 WebSocket 管理，与 DTC 项目技术栈统一 |
| **编排** | 自建状态机（初期）→ Temporal（后期） | MVP 不引入重依赖 |
| **消息** | Redis Streams（初期）→ NATS（后期） | 渐进式复杂度 |
| **数据库** | PostgreSQL 16 | 结构化数据 + JSONB |
| **缓存** | Redis 7 | 会话状态、连接状态 |
| **前端** | Next.js 15 | SSR + 实时面板 |
| **插件** | TypeScript | OpenClaw 插件必须用 TS |

## 安全架构

### Token 隔离原则
```
用户 Gateway                    平台
┌──────────────┐              ┌──────────────┐
│ [用户 Token]  │              │ [平台 Token]  │
│ [用户密钥]    │              │ [协作令牌]    │
│              │              │              │
│  绝不外传    │──协作令牌────►│  不碰用户Token │
└──────────────┘              └──────────────┘
```

### 协作令牌
- 插件安装时用户在平台注册，获得 API Key
- 加入协作时，API Key 换取短期协作令牌（24h）
- 令牌绑定 collaboration_id + permissions
- Gateway 不需要暴露端口（插件出站连接）

### 权限模型
- 平台级：owner / member / observer
- 任务级：角色能力矩阵（前端 Agent 只能操作前端文件）
- 网关侧：用户通过 `tools.allow/deny` 控制插件可用工具
